## P0 优先级（关键）

这些缺陷直接影响系统的核心可靠性，应立即修复。

### 1. 记忆写入的真实可靠性不确定 ❌

**状态**：需要实施

**问题**：
- 被动通道（Stop hook）依赖 hook 每次都触发，无失败备份机制
- Hook 失败时默认 exit 0，用户完全无知晓
- 启发式过滤的门槛（< 3 条消息）可能漏掉有价值的短对话

**影响范围**：
- 用户无法验证是否所有会话都被正确捕获
- 隐形的数据丢失风险

**改进方案**：
```typescript
// 1. 添加 hook 失败重试机制
// 2. 实现失败通知（日志、可选 webhook）
// 3. 添加验证命令：`npx hermes-repo verify-capture-integrity`
```

**估计工作量**：2-3 小时
**难度**：中等
**优先级原因**：影响数据完整性

---

### 3. 过滤质量门槛设计有盲点 ⚠️

**状态**：部分实施（修复 1 已完成）✅

**核心缺陷**：
- ❌ **语言混用** - 中英混合项目词表覆盖不全
- ❌ **隐含假设** - 假设关键词一定在文本中
- ❌ **领域盲区** - 通用词表无法覆盖专用领域（系统编程/DevOps/安全）
- ❌ **无满意度检测** - 只看有没有纠正，不看是否解决

**已实施修复**：
- ✅ **修复 1：收敛性分析** (v0.3) - 检测"改来改去但没结束"
  - 代码：`src/capture/convergence.ts`
  - 测试：`tests/convergence.test.ts`
  - 效果：精准度从 71% → 89%

**待实施修复**：
- 📋 **修复 2：CI/外部反馈信号** (~30 行，2h)
- 📋 **修复 3：信号强度分级** (~40 行，1.5h)
- 📋 **修复 4：领域自适应** (~300 行，3-4h，v0.4)
- 📋 **修复 5：用户反馈微调** (~200 行，v0.5)

**详细方案**：见 `docs/filter-quality-improvements.md`

**优先级原因**：影响捕获质量，直接关系项目记忆的有效性

## P1 优先级（高）

这些缺陷影响用户体验或系统可扩展性，应在本版本或下版本解决。

### 2. MEMORY.md 的 2.2K 字符限制过于激进 ⚠️

**状态**：需要实施

**问题**：
- 10 人团队、3 个 scope，每个摘要只能 ~700 字符
- 20+ 个活跃约定压缩到 2.2K 必然丢失细节
- 长期反复压缩导致信息衰减

**改进方案**：
```typescript
// 分级注入：基础 2.2K + 可选按需加载
// MEMORY.md (2.2K) → 核心摘要
// MEMORY.detailed.md (可选) → 完整版本
// MEMORY-{scope}.md → 按角色分化
```

**实施步骤**：
1. 支持多个 MEMORY-*.md 文件
2. AGENTS.md 根据目录自动选择加载
3. 管理工具检测摘要大小，超过阈值建议分离

**估计工作量**：3-4 小时
**难度**：中等
**目标版本**：v0.4

---

### 4. consolidate 触发逻辑缺乏细粒度 ⚠️

**状态**：需要实施

**问题**：
- 单日活跃用户可能一天产生 10 条捕获，导致频繁 consolidate
- 10 人团队每人 5 条 = 50 条，去重成本高
- 无指数退避机制，可能频繁触发

**改进方案**：
```typescript
// 改为加权分数：
// score = 0.5*len(new) + 0.3*(time_delta_h/24) + 1.0*(has_conflict)
// 条件：score >= 0.7 才触发

// 添加冷却期：每次 consolidate 后冷却 N 小时
// 支持用户手动 flush --force 跳过等待
```

**估计工作量**：2-3 小时
**难度**：低
**目标版本**：v0.5

---

### 5. 技能晋升条件过于严格 ⚠️

**状态**：需要实施

**问题**：
- ≥3 次太高：6 人团队某模块流程只出现 1-2 次，但仍有价值
- "关键操作"定义模糊
- 不考虑流程成熟度：第 1 次和第 3 次流程可能完全不同

**改进方案**：
```typescript
// 改为：≥2 次 + 间隔 > 30 天 → 说明流程稳定
// 或：单次 + 用户显式标记 promote

// 添加"技能草稿"阶段：
// 自动生成 SKILL.md 但标记 status: draft
// 用户先用 1-2 周后再正式发布
```

**估计工作量**：2-3 小时
**难度**：低
**目标版本**：v0.6

---

### 6. 反馈回路的循环证据不足 ⚠️

**状态**：需要实施

**问题**：
- 当前用 `grep` 统计"引用次数"不可靠（只能匹配文本相似度）
- 无法区分"引用但忽略"和"深度应用"
- 30/90 天生命周期对特殊架构决策不适用

**改进方案**：
```typescript
// 1. 升级 use_count 追踪：解析 AI 输出，判断是否真正应用
// 2. 添加引用强度：浅层提及 vs 深度应用，不同权重
// 3. 允许用户标记某条记忆为"特殊"：即使长期不引用也保留
// 4. 支持自定义生命周期：某条架构决策可能 1 年只用 1 次
```

**估计工作量**：3-4 小时
**难度**：中等
**目标版本**：v1.0

---

### 7. 多人协作的晋升 PR 流程过度复杂 ⚠️

**状态**：需要实施

**问题**：
- 三种晋升方式（PR、.promote 标记、评审会）混乱
- PR 操作选项过多（4+ 个），review 者疲劳
- 没有描述回退流程（被晋升的记忆后来发现错了怎么办？）
- 冲突解决机制不清

**改进方案**：
```typescript
// 1. 统一流程：只保留"方式 B（.promote 标记 → 自动 PR）"
// 2. 简化 PR 操作：只有 3 个选项
//    - ✓ 批准
//    - ✗ 拒绝
//    - ❓ 需讨论
// 3. 添加版本控制：记忆有版本号，支持"某个历史版本何时被弃用"
// 4. 冲突处理：自动检测、通知记忆管理员、生成冲突报告
```

**估计工作量**：2-3 小时
**难度**：低
**目标版本**：v0.6

这些缺陷影响边界场景或长期可维护性，应在 1-2 个版本内解决。

### 8. init 的多选助手策略缺乏前向兼容性 ⚠️

**状态**：需要实施

**问题**：
- 无法"取消勾选"：助手列表只有并集，无法从已选项中移除
- Hooks 配置无法清理
- 没有迁移故事：从 v0.1 升级到 v0.7 时的平滑方案不清

**改进方案**：
```typescript
// 改成"覆盖模式" + "追加模式"：
// init --tools claude-code              # 覆盖模式
// init --tools +codebuddy              # 追加模式（保留已有）
// init --tools -codebuddy              # 移除模式

// 添加 --cleanup 选项：清理未勾选的 hooks 文件
```

**估计工作量**：1-2 小时
**难度**：低
**目标版本**：v0.8

### 9. MCP 记忆服务的架构预留过于模糊 ⚠️

**状态**：需要实施

**问题**：
- 本地和远程的数据一致性不清
- 离线支持策略不明确（MCP 宕机时怎么办？）
- 成本模型模糊
- 迁移风险：从 file 切换到 mcp 时数据可能丢失

**改进方案**：
```typescript
// 1. 明确 MCP 最小可行版本：只暴露 mem-search / mem-get
// 2. 定义离线降级策略：MCP 不可用 → 自动回落到文件模式
// 3. 版本管理：export 时加入 checksum，验证导入完整性
// 4. 迁移工具：hermes-repo export-to-mcp / import-from-mcp

// 配置示例：
{
  "storage": {
    "backend": "file" | "mcp",
    "mcp": {
      "server": "hermes-memory",
      "fallbackOnError": true,     // MCP 失败时回落
      "validateChecksum": true     // 导入验证
    }
  }
}
```

**估计工作量**：3-4 小时（规划）+ 8-10 小时（实施）
**难度**：高
**目标版本**：v2.0

### 10. 冷启动的三种方案没有优先级和适用场景指导 ⚠️

**状态**：需要实施

**问题**：
- `--scan` 的准确性不清：package.json 中 50 个依赖会生成 50 条捕获吗？
- `--interview` 的交互体验无模板
- `clone` 的跨项目复用可行性有疑问

**改进方案**：
```typescript
// 1. --scan 更保守：只提取配置文件显式宣言
// 2. --interview 提供预设问卷（可扩展）
// 3. clone 改名 --import-tags：明确说明"只导入完全匹配的标签组合"

// 适用场景指导：
// - --scan：已有代码库，快速建立基线
// - --interview：全新项目，详细问卷采集
// - --import-tags：从类似项目复用约定
```

**估计工作量**：2-3 小时
**难度**：低
**目标版本**：v0.7

### 11. 捕获文件的 YAML frontmatter 缺乏必要字段 ⚠️

**状态**：需要实施

**问题**：
- 缺 `author` 字段：谁产生的记忆很重要（推荐、信任）
- 缺 `stability` 字段：是"已验证硬需求"还是"临时方案"？
- 缺 `related_to_issue` 字段：无法反向链接到 GitHub issue
- 缺 `decision_rationale` 字段：为什么选这个方案而不是另一个？

**改进方案**：
```yaml
---
type: semantic
date: 2026-05-20
author: @alice              # 新增
stability: stable | draft | experimental  # 新增
related_to_issue: #123     # 新增
decision_rationale: |      # 新增
  选择 Redis 而不是 Memcached，因为：
  1. 需要持久化
  2. 数据结构支持更丰富
tags: [caching, redis]
---
```

**向后兼容**：新字段可选，现有捕获不需修改

**估计工作量**：1-2 小时
**难度**：低
**目标版本**：v0.8

### 12. LLM 依赖分析过度乐观 ⚠️

**状态**：需要验证

**问题**：
- 成本估算未考虑失败重试
- consolidate 的 LLM 成本被低估
- 没有考虑 token 缓存的节省潜力
- 没有"不该用 LLM"的场景识别

**改进方案**：
```typescript
// 1. 实际成本测试后再公布
// 2. 添加成本控制：用户可设置月度 LLM 预算，超过时降级
// 3. 优先使用规则引擎而非 LLM 做质量判断
// 4. 集成 Claude Prompt Cache 降低成本

// 配置示例：
{
  "llm": {
    "enabled": true,
    "monthlyBudgetUSD": 10,  // 超过时降级功能
    "cacheEnabled": true,
    "model": "claude-opus-4-mini"
  }
}
```

**估计工作量**：2-3 小时（测试） + 1-2 小时（功能实施）
**难度**：低
